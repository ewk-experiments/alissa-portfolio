IBM Cloud File Storage Encryption in Transit

Reduced user steps from 13 to 5 (61.5%) compared to initial proposal through service blueprinting, concept validation, and iterative design.

Lead Product Designer

Skills

Timeline

Company

IBM Cloud

Context

IBM Cloud File Storage allows users to provision network-attached storage volumes for their Virtual Server Instances (VSIs). Encryption in transit (EIT) ensures data is encrypted as it moves between the VSI and the file storage volume -- a critical security requirement for enterprise customers.

Challenge

The initial engineering proposal for enabling encryption in transit required users to complete 13 manual steps, including certificate management, command-line configuration, and ongoing certificate renewal. This was a borderline-unusable experience that would generate significant support burden and user frustration.

Service blueprint mapping the initial 13-step user flow

Mapping the User Flow

Started by facilitating workshops with engineering, product management, and security teams to map the complete end-to-end user flow. Created service blueprints that revealed the full complexity of what users would face -- not just the provisioning step, but the ongoing maintenance burden.

Discovering the Borderline-Unusable Experience

The service blueprinting process exposed several critical issues beyond the initial setup complexity:

  • Certificate renewal was required every 90 days -- and failure to renew would silently break the encrypted connection
  • Users had no visibility into certificate expiration status
  • The manual mount process was error-prone and varied across operating systems
  • Documentation alone could not compensate for the complexity of the workflow

Pivoting to Mount Helper Utility

Rather than trying to make 13 steps more usable, advocated for a fundamentally different approach: a mount helper utility that would automate certificate management, mounting, and renewal behind the scenes. This shifted the experience from manual CLI operations to a simple toggle in the provisioning UI.

Before and after comparison: 13 manual steps vs 5 streamlined steps

Concept Validation

Conducted 11 concept validation interviews with target users (cloud infrastructure administrators and developers) to test the simplified approach. The interviews confirmed that:

  • Users strongly preferred the automated approach over manual certificate management
  • The provisioning toggle was immediately understood
  • Automatic certificate renewal removed a major anxiety point
  • The mount helper utility concept aligned with existing mental models

Final UI

The final design reduced the user experience to 5 clear steps: provision a file storage volume with EIT enabled (via toggle), attach it to a VSI, and the mount helper handles certificates and encryption automatically. Certificate renewal happens in the background with no user intervention required.

Final provisioning UI with encryption in transit toggle

Outcomes

61.5% fewer steps
Reduced from 13 to 5 user steps
6+ months
Technical debt prevented by catching issues early

The project received product awards for its user-centered approach to a deeply technical problem. Launched in August 2023.

Reflection

This project reinforced the value of "making to think" -- using service blueprints and flow mapping not just as documentation tools, but as thinking tools that reveal hidden complexity. The 13-step flow looked reasonable in a requirements document. It was only when mapped end-to-end, including maintenance and renewal flows, that the experience became clearly unacceptable.

By catching this early through design-led investigation, the team avoided shipping a feature that would have created months of technical debt and user support burden.
